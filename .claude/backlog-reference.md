# Backlog Execution Reference

Shared constants and patterns for the `/backlog:*` commands. Generated by `/backlog:setup`.

## Status Labels

Commands use `status:*` labels on issues for pipeline tracking. Created by `/backlog:setup`.

| Label | Description |
|---|---|
| `status:assess` | Awaiting scope classification |
| `status:plan` | Awaiting implementation plan |
| `status:plan-review` | Plan posted, awaiting review |
| `status:implement` | Ready for implementation |
| `status:monitor` | PR open, CI being watched |
| `status:merge` | CI passed, awaiting merge |
| `status:blocked` | Agent blocked, needs intervention |
| `status:done` | Completed |

## Scope Classification

Configuration for `/backlog:assess` scope heuristics. Edit these to match your project's conventions.

### Small Scope Indicators

Labels that suggest small work:
- `bug`
- `documentation`

### Large Scope Indicators

Labels that suggest large work:
- `enhancement`
- `improvement`

Keywords in the issue body that override to Large:
- `migration`
- `schema`
- `new table`
- `RLS`
- `new component`
- `new API route`
- `new endpoint`

### Defaults

- **Default scope:** `Large` (when no indicators match; typically Large)
- **Fix-provided override:** `true` (if true, an explicit fix in the issue body classifies as Small regardless of other signals)

## Common `gh` Patterns

### Check an issue's current status

```bash
gh issue view <NUMBER> --json labels --jq '[.labels[].name | select(startswith("status:"))] | first'
```

### Transition an issue's status

```bash
gh issue edit <NUMBER> --add-label "status:NEW" --remove-label "status:OLD"
```

### Post a comment on an issue

```bash
gh issue comment <NUMBER> --body "comment text"
```

### Read full issue details

```bash
gh issue view <NUMBER> --json title,body,labels,comments
```

## Worktree Convention

- Directory: `.claude/menehune/worktrees/agent-<number>-<slug>`
- Branch: `agent/<number>-<slug>`
- Example: `.claude/menehune/worktrees/agent-85-rls-fix`, branch `agent/85-rls-fix`

All menehune runtime state lives under `.claude/menehune/` (worktrees, locks) so it stays out of the consumer's own `.claude/` tree and doesn't clash with other plugins.

## File-Path Denylist

Before creating a PR, check `git diff --name-only` against these patterns. If any match, abort to "Blocked":

- `.env*` - environment secrets
- `package-lock.json` - only modify via npm install
- `bundle/` - MCP bundle artifacts
- `*.mcpb` - MCP bundle packages

## Agent Work Locking

Commands that create or modify worktrees (`/backlog:plan`, `/backlog:implement`) use lock directories to prevent parallel work conflicts:

- Lock directory: `.claude/menehune/locks/NUMBER.lock/` (created atomically with `mkdir`)
- Metadata file: `.claude/menehune/locks/NUMBER.lock/info` (contains command name and timestamp)
- Created at command start, removed on completion (`rm -r`)
- Stale lock detection: locks older than 4 hours are auto-removed (likely from crashed sessions)
- Fresh locks (<4 hours): command warns and asks whether to proceed
- Cleanup order: lock is always removed last, after all other resources

If `mkdir` gets denied by the Claude Code permission layer, the user has not run `/backlog:setup` (or a recent `/backlog:setup refresh`) -- the lock-directory wildcards are added to `settings.local.json` at setup time. Commands fall back to skipping the lock step and log a warning, but batch runs lose the conflict guard.

## CE Plugin Dependency

Commands that delegate to Compound Engineering skills (`/backlog:plan`, `/backlog:implement`) require the CE plugin. The commands check for skill availability at runtime and provide inline fallback instructions if unavailable.
