## Known gotchas (carried — still apply to any future work)

- **Path interpolation needs an allowlist.** Any value interpolated into a request path (string `field_code`/keys; integer IDs via `z.number().int().positive()` are safe) must use a strict allowlist regex, NOT a `/`-only blocklist — `new URL()` normalizes `\`, `..`, `?`, `#`. (See #70 P1 above.)
- **MERGE-GATE GOTCHA.** Do NOT use `gh pr merge <n> --squash --delete-branch` (its local-branch delete fails because the **worktree** holds the branch, and it skips the remote delete). Use **`gh pr merge <n> --squash` WITHOUT `--delete-branch`**, then manual cleanup from the MAIN repo root: `git worktree remove <path> --force` → `git branch -D <branch>` → `git push origin --delete <branch>` → `git worktree prune` → `rm -r .claude/menehune/locks/<n>.lock` → `git checkout main && git pull --ff-only`. Verify with `gh pr view <n> --json state,mergedAt`, `git worktree list`, `git ls-remote --heads origin <branch>` (don't trust exit codes).
- **MERGE-DIVERGENCE GOTCHA (new, #70).** If you rebase an agent branch onto a LOCAL `main` that is ahead of `origin/main` (e.g. an unpushed handoff-doc commit), those unpushed commits ride into the pushed PR branch as ancestors and the GitHub squash folds them into the merge commit. After merge, `git pull --ff-only` then FAILS ("Not possible to fast-forward") because local/origin diverged at the shared base. Fix: confirm the unpushed change is contained in the squash (`git diff <local-commit> origin/main -- <file>` is empty), then `git reset --hard origin/main`.
- **Adding a param means editing THREE places:** the Zod schema, the hand-written `inputSchema` JSON literal (NOT generated from Zod), and the handler's query/body/path forward line.
- **`git checkout <ref> -- src/` does NOT delete files absent from `<ref>`** — for a complete revert-proof, `rm` new files explicitly, then `git checkout origin/main -- <modified existing files>`.
- **Verify LLM-facing tool-description text against the actual Zod INPUT schema, not the API query key.** (#68 R2 caught a description saying `project_id` when the input field is `id`.)
- **Auth reality (supersedes CLAUDE.md prose):** v2 uses the **`x-api-token` HEADER** (`src/client.ts` `applyAuth`), NOT the `api_token` query param (v1-only).
- **`.env` is auto-loaded by anything importing `src/index.ts`** (it does `import "dotenv/config"`), so dev scripts/tests that import the server inherit the repo's gitignored `PIPEDRIVE_API_KEY` as a FALLBACK. An explicit shell `PIPEDRIVE_API_KEY=…` overrides it (dotenv never overwrites an already-set var). The #76 smoke harness relies on this but guards full mode (which CREATES/DELETES real records) behind a required `--confirm-sandbox` flag plus a masked token-tail print, so it cannot silently run writes against whatever account `.env` points at. Re-apply that pattern for any future live-API script.
- **Lint IS a CI gate** (`.github/workflows/ci.yml` runs `npm run lint` = `eslint src/`; test files NOT linted). Verify with `npx eslint src/`.
- **Zod 4.4.3 on main** (NOT Zod 3). Idioms: `z.email()`/`z.uuid()`, `z.record(z.string(), z.unknown())`, `.extend(...shape)` (not `.merge`), `error.issues` (not `.errors`). Zod `.refine({path})` wants a MUTABLE `PropertyKey[]` — do NOT `as const` the options object (a readonly `path` fails `tsc`).
- **`createListSummary` naively appends "s"** for plurals — assert summaries with `toContain`, not exact `toBe`.
- **False-green hazard:** `tsconfig` excludes `**/*.test.ts` (tests never type-checked) and per-entity tests call handlers DIRECTLY (bypass the dispatcher's Zod). Real guards = UNIT schema tests (`Schema.parse`, `.strict()` REJECT tests) + integration tests asserting the wire shape (`JSON.parse(options.body)` / the URL) + the `tests/contract/` OpenAPI harness.
- **Contract harness handles array bodies + composed schemas (#75).** `tests/contract/helpers/openapiContract.ts` now exposes `assertArrayBodyConformsToSpec(opId, body)` for ops whose v2 request body is a top-level JSON array (the field-option bulk verbs); the object-only `assertBodyConformsToSpec` rejects non-objects by design. `requestBodyProps`/`requestBodyPropType` merge top-level `allOf` via `collectProperties`, so `allOf`-composed bodies (`addProduct`/`updateProduct`) resolve their keys instead of returning `[]`. CAVEAT: the body-shape check only sees the keys a handler actually emits, and contract tests bypass Zod — so it CANNOT prove a narrower schema (e.g. person/product field bodies excluding `description`); that exclusion is the Zod `.strict()` schema's job. The array assertion is shape-only: no `minItems`, so an empty array passes, and it skips item key-checks when the spec `items` declares no `properties`.
- **Test layout:** unit schema tests in `tests/unit/schemas/` (3-level import), integration handler tests in `tests/integration/tools/`, helpers at `tests/helpers/`. **`setupValidEnv()` sets `PIPEDRIVE_ENABLE_DESTRUCTIVE='true'` in `beforeEach`** — "enabled" delete tests work by default; "guard blocks" tests must `delete process.env.PIPEDRIVE_ENABLE_DESTRUCTIVE` explicitly.
